本文是《十年后端，大模型时代的重新定位》的写作规格文档。

## 发布渠道

完整版发布到 GitHub 个人站、知乎和公众号。

小红书版本等完整版定稿后，再拆分为多篇笔记；当前阶段不提前规划具体标题和提纲，避免影响完整版结构。

## 读者定位

核心读者是正在求职或准备跳槽的后端开发，尤其是有一定经验、以 Java / Go 等技术栈为主、没有明显大厂或算法背景的人。

外延读者是所有关心 AI 时代后端定位变化、想提前补齐大模型应用能力的后端开发。

## 文章定位

主线是一次后端求职复盘。

副线是从这次经历中提炼出普通后端在 AI 时代的转向建议。

文章不是宏观就业市场分析，也不是大模型应用后端学习路线、RAG / Agent 教程或面试题库。技术内容只服务于求职复盘，写到能解释岗位匹配和准备方向即可。

## 核心结论

这篇文章的核心判断是：AI 时代后端岗位的变化，不只是岗位数量变化，也不是传统后端内容消失，而是后端基础能力仍然重要，但 AI 相关能力在面试和工作方式里的比重正在增加。

传统后端能力的考察没有消失，也不应写成被 AI 取代。数据库、并发、系统设计、稳定性、线上问题排查等传统后端内容仍然是基础。

在传统后端能力之上，AI Coding 正在成为新的基础能力。很多后端岗位已经默认候选人会使用 AI Coding，并会关注候选人是否有实际使用经验，能不能说明具体使用场景、约束生成质量、审查代码和验收结果的方式。

大模型应用能力是另一个层级：做过 RAG / Agent 相关项目，能讲清楚大模型应用从场景、流程、选型到问题处理和评估的完整过程，对于普通后端是明显加分项；进一步发展，则可以对应到“大模型应用后端 / 大模型应用全栈”这类岗位。

从个人求职经历看，普通后端想靠近 AI，不一定要转算法或纯 AI 开发。更适合的方向，是把已有后端工程能力用到大模型应用开发中，优先关注“大模型应用后端 / 大模型应用全栈”这类岗位。

这不是说所有后端都必须转向大模型应用后端，也不是说传统后端没有价值。只是从这次求职经历看，如果一个普通后端想靠近 AI，又不具备算法和模型背景，那么大模型应用后端是一条更现实的路径。

## 表达边界

文章基于个人经历和阶段性判断，不写成确定性的行业结论。

不要渲染焦虑：

- 不使用“后端已死”“不转 AI 就淘汰”这类表达。
- 不把 AI 描述成单向取代程序员，而是描述成改变岗位要求和开发方式。
- 不贬低传统后端经验，而是强调后端经验在 AI 应用落地中的迁移价值。
- 结尾落到可操作建议和重新定位，而不是焦虑情绪。

关于岗位变化，重点不是写普通后端岗位减少，也不是写 AI 简单取代后端，而是写传统后端内容仍然是基础，AI 相关能力在面试和工作方式里的比重正在增加：传统后端能力仍然要考察；AI Coding 逐渐成为基础能力；大模型应用项目经验和业务落地能力会成为重要加分项。公司对“纯业务开发 / 纯编码执行”的需求更谨慎，对候选人的综合能力要求更高。

关于 AI 和后端职责，不要写成“AI 做不了系统设计 / 技术选型 / 问题排查”。更稳妥的表达是：AI 已经能参与编码、方案讨论、技术选型、问题排查等工作，但在真实项目里，后端仍然要对系统结果负责。比如需求边界是否清楚、方案是否适合当前业务、并发和稳定性风险有没有覆盖、线上问题如何闭环、AI 生成的代码能不能上线，这些判断和责任目前仍然离不开开发者。

## 开头要求

开头不要以“工作不好找”“普通后端岗位变少”这类大环境共识作为第一句，也不建议在开头主动强调岗位减少。这类表达在当前环境下信息增量不够，容易显得像废话。

第一句应从“求职复盘后的判断”切入，而不是从具体时间点切入。避免使用“年初找工作时”作为开头，因为这会让读者感觉事情已经过去一段时间，削弱文章的当下价值。

第一句不必承担完整结论，但要提出一个有吸引力、能让目标读者继续读下去的核心观察：这次找工作让我体会到，对于普通后端来说，AI Coding 和大模型应用能力，确实能明显带来更多面试机会。

这样设计第一句的原因是：文章的开头首先要让目标读者产生继续读的理由，而不是一上来写成摘要式结论。完整结论可以写入正文第一段或后续段落，但第一句应优先建立文章张力：这不是泛泛讨论就业环境，也不是制造“后端已死”的焦虑，而是从一次求职复盘中提出一个对普通后端有实际参考价值的观察。

第一句应避免泛泛说“懂 AI”“往 AI 靠”，也不要只写成“考察范围变宽了”这类抽象判断。完整判断可以放在第一段内展开：传统后端基本功仍然是基础，但 AI Coding、RAG / Agent 和大模型应用落地能力正在更频繁地影响岗位匹配和面试评价。

开头应尽快引出核心问题：AI 时代后端岗位的默认能力正在扩展，AI Coding 正在成为基础能力，大模型应用项目经验和落地能力则是重要加分项，并可能进一步发展为“大模型应用后端”方向；后端需要重新理解自己的定位。

开头控制篇幅，避免反复铺陈求职困难。个人背景、求职体感和核心结论要尽快进入主线。

## 文风要求

文章定位为技术经验复盘，语气要克制、清晰、具体。

可以保留个人经历和第一人称，但避免过度口语化、情绪化和泛泛表达。少用“没以前好找了”“压哨”“硬冲”“往 AI 靠”等偏口语词，除非确实能增强表达。

正文也要少用概念腔、评论腔较重的词，尤其要避免常见“互联网黑话”。大多数技术开发人员不喜欢这类表达，因为它容易显得浮夸、空洞，不如具体技术问题和真实经历有说服力。

少用或慎用的词包括：“基本盘”“完整链路”“业务链路”“抓手”“赋能”“闭环”“兜住”“收敛到”“提效”“打法”“心智”“沉淀”“颗粒度”“生态”等。如果不是必要技术表达，应优先换成更朴素、具体的说法，例如“基础”“完整过程”“业务流程”“控制风险”“重点放到”。“落地”这类词可以在讨论工程实践时少量使用，但不要用它替代具体动作和问题。

正文里也慎用“复盘”这类泛词，要看场景。需要表达文章来源时，可以写“这次求职经历”“这次找工作”“我这次面试下来”等更具体的说法。

虽然文章不是纯技术教程，但仍然应按技术文章的标准写：语言精炼、准确，不用大词替代事实，不用抽象判断替代具体例子。

优先使用具体观察、面试问题、岗位要求变化和真实技术问题来支撑观点。

## 岗位分类口径

正文保持两类讨论：

- AI 开发：包含更偏算法、模型、NLP、微调、训练、评估的岗位，也包含一些名为“大模型开发”的岗位。这类“大模型开发”可能并不直接做模型训练，但会更专门地做复杂 RAG / Agent 系统、大模型能力开发、评估和调优，对 AI 相关背景、项目深度和技术深度要求更高。
- 大模型应用后端：以业务系统和工程落地为主，叠加 RAG、Agent、AI Coding 等能力。

现实 JD 里的“大模型开发 / 大模型应用开发 / AI 应用开发 / 大模型全栈”等名称边界很模糊，可能同时包含应用开发、后端、前端和部分算法能力。文章重点不是做岗位分类，而是说明传统后端更容易迁移到哪一类工作内容。

需要明确区分“大模型开发”和“大模型应用后端”。大模型开发在本文里更接近 AI 开发，通常对学历、背景、项目深度要求更高；大模型应用后端首先仍然是后端岗位，重点是把 RAG / Agent 等能力放进业务系统里，并处理接口、数据、权限、部署、稳定性等后端问题。正文应写出作者在这里曾经困惑过，并说明判断方式：不要只看岗位名，要看 JD 里的实际要求。

工具调用属于大模型应用和 Agent 开发里的基础知识，不应和 RAG、Agent 并列成同一层级。正文可以在讲 Agent 工程问题时提到工具调用、失败处理、状态流转等内容。

## 关键观点

### 后端岗位要求变化

这次找工作最明显的体感是，后端基础能力的考察没有变，但对 AI 相关能力考察的比重正在增加。传统后端内容没有减少，后端基本功（八股）仍然是考察重点。在此基础上，很多岗位会增加 AI Coding 相关考察，关注候选人是否有实际使用经验，能不能说明具体使用场景、约束生成质量、审查代码和验收结果的方式。

大模型应用能力需要和 AI Coding 区分开：AI Coding 更像普通后端也需要补齐的新基础能力；RAG / Agent 相关项目经验，以及大模型应用的完整落地能力，则更像加分项和进一步的岗位分支。

### 传统后端经验仍然有迁移价值

后端不是从零开始转 AI。大模型应用最后仍然要落到业务系统里：权限、数据、流程、接口、任务调度、监控、部署、稳定性、成本控制，这些都离不开后端经验。

所谓转向大模型应用后端，本质是把原来的工程经验迁移到新的业务形态里。

### AI 开发不适合普通后端短期直接转

可以讲自己也尝试过更偏 AI 开发的岗位，但要控制篇幅，并保持提醒语气。

主要原因是学历门槛、项目经验、知识背景不匹配。很多岗位会看算法、模型、NLP、微调、训练、评估等背景；部分“大模型开发”岗位虽然做的是 RAG / Agent，但项目复杂度和专业深度也更高。这些和传统后端工程经验不是同一类积累。对普通后端来说，短期直接投这类岗位，投入产出比可能并不理想。

这段不要写成失败叙事，也不要写成对岗位的批判，而是提醒读者不要只看岗位名，要看 JD 和自身匹配度。

### 大模型应用后端首先仍然是后端岗位

大模型应用后端不是只会调 API，也不是只会写 Prompt，而是在后端系统中接入和落地 AI 能力。

这类岗位仍然考察后端基本功，不因为岗位带 AI 就降低要求。AI 相关能力是在后端能力之上的补充，而不是对后端能力的替代。

### 能力判断只服务于方向判断

文章重点是说明普通后端为什么更适合关注大模型应用后端，而不是讲完整的学习路线或操作指南。能力相关内容只作为方向判断的支撑，写到“这类岗位会看什么、为什么这些能力和后端有关”即可，不展开成系统学习路线，也不把文章写成“怎么做”的教程。

重点保留三组：

- 后端基本功不能丢：面试仍然会考后端，不因为岗位带 AI 就降低要求。
- RAG / Agent 要有项目感：不仅知道概念，还要能讲流程、选型、问题处理。
- AI Coding 要会落地：能说明自己在哪些场景使用 AI Coding，怎么约束质量、怎么审查代码、怎么验收结果。

### 补齐项目经验

这部分有必要，但只写一小段。

如果没有真实公司项目，至少要准备一个小而完整的 RAG / Agent 项目，用来证明自己做过完整项目。重点不是项目多大，而是能讲清楚场景、流程、选型、问题和评估。

不展开实现细节。

### 面试问题只做经历佐证

面试问题不要整理成题库，也不要归类成系统面试指南。

保留“我被问过这些问题”的口吻，用少量真实问题说明：这类岗位更看重项目经验、场景判断和排查思路，而不是背概念。

可保留的问题方向：

- Agent 怎么避免死循环？
- 工具调用失败怎么处理？
- RAG 效果不好时从哪里排查？
- AI Coding 用过哪些 Skills，分别在什么场景解决了什么问题？

### AI Coding 是独立但次一级的必备能力

AI Coding 不是文章主线，但要作为一个明显的小节。

需要表达几个点：

- 面试过程中逐渐发现，很多公司已经默认候选人会使用 AI Coding。
- 被问的重点不是“你用不用工具”，而是“你怎么用、在哪些场景用、解决过什么问题”。
- 入职后进一步确认，AI Coding 已经成为实际开发中的主要工作方式之一。
- AI Coding 不等于让 AI 写代码，而是人负责拆任务、定边界、做架构判断、审查代码和验收结果。
- AI 仍然会遗漏工程问题，例如集群部署并发、资源争抢、边界条件、质量控制等。

## 求职细节边界

写体感和阶段性结论，不写具体数字。

可以写：

- 年初找工作。
- 后端基础能力的考察没有变，传统后端内容仍然会考，对 AI Coding 相关内容考察的比重正在增加。
- 大模型应用能力对于普通后端是加分项，也可以进一步对应到大模型应用后端岗位。
- 尝试过 AI 开发方向但匹配度不高。
- 后来把重点放到大模型应用后端。
- 面试中发现后端基本功、RAG / Agent 项目经验、AI Coding 都会被关注。
- 五一前入职。

不建议写：

- 投了多少份简历。
- 收到多少面试。
- 哪些公司。
- 薪资变化。
- 具体 offer 数量。
- 过于细的失败经历。

## 建议结构

1. 开头从求职复盘后的核心观察切入：对普通后端来说，AI Coding 和大模型应用能力确实能明显带来更多面试机会。随后补足完整判断：后端基础能力仍然重要，传统后端内容仍然会考，但岗位匹配和面试评价中 AI 相关能力的比重正在增加。
2. 引出文章的问题：一个普通后端在 AI 时代应该怎么重新定位。这里可以交代个人背景和求职经历，但不要把重心放在岗位减少或求职困难上。
3. 先区分几个容易混在一起的概念：AI Coding、AI 开发、大模型应用后端。重点不是做严格分类，而是让读者理解自己接下来讨论的是哪一种转向路径。
4. 说明自己曾经观察或尝试过 AI 开发方向，但学历门槛、项目经验和知识背景匹配度不高。该段用于解释为什么不建议普通后端短期直接转纯 AI 开发岗位。
5. 转到大模型应用后端：它更能复用后端经验，也能进入 AI 应用开发场景，是普通后端靠近 AI 更适合的方向。
6. 解释大模型应用后端不是只调 API，而是在后端系统里落地 AI 能力；同时强调这类岗位首先仍然是后端岗位，后端基本功不会因为岗位带 AI 而降低要求。
7. 简要说明这类岗位会关注的能力：后端基本功、RAG / Agent 项目感、AI Coding。能力内容只服务于解释方向选择，不展开成系统学习路线或具体操作指南。
8. 用少量真实面试问题作为佐证，不写成题库。问题要服务于观点：这类岗位更看重项目经验、场景判断和排查思路。
9. 在 AI Coding 小节加入入职后的验证：它已经成为实际开发中的主要方式之一。重点写清楚 AI Coding 不是让 AI 代替程序员，而是人负责拆任务、定边界、做架构判断、审查代码和验收结果。
10. 结尾回扣“重新定位”：不是从后端跳到算法，也不是只学几个框架，而是把已有工程经验迁移到 AI 应用落地中，同时补齐新的应用能力。

## 结尾方向

结尾要接应标题里的“重新定位”。

推荐落点：对普通后端来说，AI 时代不是简单地从后端跳到算法，也不是只靠多学几个框架就能解决问题。更重要的是重新理解自己的位置：把已有的工程经验迁移到 AI 应用落地里，同时补齐 RAG、Agent、AI Coding 这些新能力。

最后保留个人边界：这只是这次求职后的阶段性判断，但至少对自己来说，它比硬转纯 AI 开发更现实。

## 读者评审流程

完整版初稿完成后，可以让 AI 以读者视角进行评审。

读者评审采用 2+1 设计：两个必选目标读者，一个终稿前专业审稿人。

- 初稿和大改后：使用读者 A、读者 B。
- 准备发布前：使用读者 A、读者 B、读者 C。
- 读者 C 只做专业性兜底，不决定文章主线。

### OpenRouter 评审方式

读者评审优先通过 OpenRouter API 调用不同大模型执行。

主 agent 负责从本文 spec 中裁剪当前读者需要的评审规则，并为每个读者生成一组 `system` 提示词和 `user` 提示词。裁剪内容包括：当前评审阶段、共用检查项、AI 味检查标准、当前读者角色定位、当前读者主要判断、当前读者重点打分、当前读者不应该要求、反馈格式、当前读者相关通过标准、评审边界和唯一文章路径。

`system` 提示词用于固定当前读者身份、评审任务和行为约束。它应该包含当前读者的角色定位、主要判断、重点打分、不应该要求、评审边界，以及“不重写全文、只输出评审结果、不做采纳决策”的要求。

`user` 提示词用于提供当前评审阶段、共用检查项、AI 味检查标准、反馈格式、当前读者相关通过标准，以及文章正文。文章正文 `content/posts/backend-ai-era.md` 是唯一评审对象；如果调用端不能让模型读取本地文件，应由主 agent 将正文内容放入 `user` 提示词。

每个读者角色可以调用 2-3 个不同模型进行独立评审。不同模型使用同一组 `system` / `user` 提示词，避免因为提示词差异导致反馈不可比较。主 agent 不向模型提供其他读者的完整定义、其他模型的评审结果，或主 agent 对文章质量的预判。

模型只输出评审结果，不做采纳决策；是否修改正文由主 agent 汇总后再由作者确认。汇总时应区分多模型共识问题、单模型个别意见和模型之间的明显分歧。

### OpenRouter 调用约定

调用 OpenRouter 时使用 Chat Completions API。

基础配置：

- API 地址：`https://openrouter.ai/api/v1/chat/completions`
- 鉴权方式：通过环境变量 `OPENROUTER_API_KEY` 提供 API key，请求头使用 `Authorization: Bearer <OPENROUTER_API_KEY>`。
- 请求格式：使用非流式请求，`messages` 中只包含一条 `system` 消息和一条 `user` 消息。
- 生成参数：评审任务优先保持稳定输出，`temperature` 建议设置为 `0.2` 左右；输出长度按模型能力设置，避免因为截断导致评审不完整。
- 模型列表：由调用端通过变量配置，每次评审至少选择 2 个风格不同的模型，终稿前可增加到 3 个。
- 默认模型变量：`OPENROUTER_REVIEW_MODELS=["anthropic/claude-opus-4.7","google/gemini-3.1-pro-preview","qwen/qwen3.7-max"]`。后续如果模型不可用、成本过高或评审质量不稳定，可以只调整变量，不修改评审流程。

`system` 提示词模板：

单次请求结构示例：

```json
{
  "model": "<从 OPENROUTER_REVIEW_MODELS 中取出的 model id>",
  "messages": [
    {
      "role": "system",
      "content": "<当前读者角色的 system 提示词>"
    },
    {
      "role": "user",
      "content": "<当前评审阶段、检查项、通过标准、反馈格式和文章正文>"
    }
  ],
  "temperature": 0.2,
  "max_completion_tokens": 3000
}
```

执行流程：

1. 主 agent 读取本文 spec 和文章正文。
2. 按当前阶段选择读者角色：初稿和大改后使用读者 A、读者 B；终稿前增加读者 C。
3. 为每个读者角色生成固定的 `system` / `user` 提示词。
4. 用同一组提示词分别调用多个模型。
5. 保存每个模型的原始评审结果，至少记录读者角色、模型名称、评审时间和输出内容。
6. 主 agent 汇总评审结果，按“共识问题 / 分歧问题 / 可忽略意见 / 建议修改项”整理。
7. 是否修改正文由作者确认，不由模型自动决定。

失败处理：

- 某个模型调用失败时，可以重试一次；仍失败则记录失败模型和原因，继续使用其他模型结果。
- 如果模型输出不符合反馈格式，可以要求同一模型基于原输出重新整理格式；不要让它重新评审文章。
- 如果不同模型结论冲突，优先保留为分歧点，由主 agent 结合文章定位和通过标准判断，不直接按多数票修改。
- 不把 API key、完整请求头、个人本机路径或包含凭据的信息写入项目文档。

### 共用检查项

所有读者都需要关注：

- 文章是否制造“后端不转 AI 就会被淘汰”的焦虑。
- 文章是否围绕求职复盘和重新定位展开，而不是变成教程、题库或学习路线。
- 文章是否存在明显 AI 味。

AI 味重点检查以下信号：

- 模板化结构明显，段落像通用文章生成模板。
- 泛化判断过多，缺少个人经历、面试问题或真实场景支撑。
- 求职复盘的现场感不足，看不到作者判断变化的过程。
- 表达过度平滑，只有正确结论，缺少具体取舍。
- 热词堆叠，RAG / Agent / AI Coding 等概念没有落到场景。
- 缺少作者自己的犹豫、试错和修正过程。

如果 `AI 味控制` 低于 4/5，读者必须指出最像 AI 生成内容的 1-3 个具体位置，并说明原因。

### 读者 A：初中级后端读者（必选）

角色定位：

- 约 3 年后端经验，以 Java / Go 等常见后端技术栈为主。
- 已经听说 AI Coding、RAG、Agent 等概念，但缺少真实项目经验。
- 对“后端是否会被 AI 淘汰”“是否必须转型 AI”存在困惑。
- 希望从资深后端的求职复盘中获得方向感，而不是完整学习路线。

主要判断：

- 文章是否能说明传统后端能力仍然有价值。
- 文章是否能给出靠近 AI 的现实路径，而不是制造“必须转型”的压力。
- 文章是否能区分 AI Coding、大模型应用能力、AI 开发岗位。
- 文章是否有足够的可执行启示，但没有变成教程或路线图。

重点打分：

- 方向启发：是否能帮助初中级后端理解靠近 AI 的现实路径。
- 焦虑控制：是否避免“后端不转 AI 就会被淘汰”的压迫感。
- 理解门槛：是否能读懂关键判断，不要求懂所有技术细节。
- 可执行感：是否知道下一步大概该补什么。
- AI 味控制。

不应该要求：

- 不要求文章解释所有基础概念。
- 不要求文章展开完整学习路线。
- 不要求文章回答“3 年经验如何拿 offer”的具体策略。

### 读者 B：资深后端读者（必选）

角色定位：

- 5-10 年后端经验，以 Java / Go 等常见后端技术栈为主。
- 有一定系统设计、线上稳定性、业务落地和项目负责经验。
- 正在求职、准备跳槽，或正在思考 AI 时代后端定位变化。
- 是本文最核心的同类读者。

主要判断：

- 文章是否贴合资深后端的求职和定位处境。
- 文章是否尊重传统后端经验，而不是把传统后端写成过时能力。
- 文章是否讲清楚后端工程能力如何迁移到大模型应用落地。
- 文章是否像真实求职复盘，而不是泛泛职业建议。

重点打分：

- 处境贴合：是否贴合 5-10 年后端的处境。
- 经验迁移：是否讲清楚传统后端经验如何迁移到大模型应用落地。
- 边界克制：是否没有夸大 AI、贬低传统后端或强行劝转。
- 复盘真实感：是否像真实求职复盘，而不是泛泛职业建议。
- AI 味控制。

不应该要求：

- 不要求文章变成大模型应用后端学习路线。
- 不要求文章展开完整面试题库。
- 不要求为了技术完整性牺牲个人复盘主线。

### 读者 C：行业 / 技术专家审稿人（终稿前使用）

角色定位：

- 有后端工程经验，也了解大模型应用落地。
- 熟悉 RAG、Agent、AI Coding 等实践。
- 不作为目标受众，而是专业性和正确性的审稿人。

主要判断：

- AI 开发、大模型应用后端、AI Coding 的边界是否清楚。
- RAG / Agent / AI Coding 等技术表达是否准确。
- 文章是否存在概念堆砌、技术误导或过度简化。
- 文章是否符合真实工程落地经验。

重点打分：

- 技术准确性：RAG / Agent / AI Coding 等表达是否准确。
- 岗位边界：AI 开发、大模型应用后端、AI Coding 的区分是否清楚。
- 工程真实感：是否符合真实项目落地经验。
- 概念克制：是否避免概念堆砌和过度展开。
- AI 味控制。

不应该要求：

- 不要求文章面向专家读者重写。
- 不要求文章补充大量技术细节。
- 不要求把文章改成教程、技术方案或白皮书。

### 通过标准

- 两个必选读者的 `AI 味控制 < 4/5` 时，必须修改。
- 读者 A 的 `焦虑控制 < 4/5` 时，必须修改。
- 读者 B 的 `处境贴合 < 4/5` 或 `经验迁移 < 4/5` 时，必须修改。
- 终稿前读者 C 的 `技术准确性 < 4/5` 或 `岗位边界 < 4/5` 时，必须修改。
- 单项 3/5 不一定全盘否定，但需要指出具体问题位置和原因。

### 评审边界

读者评审角色只负责反馈问题，不直接重写全文。

允许：

- 指出具体问题位置。
- 说明为什么影响目标读者、技术可信度或阅读体验。
- 提供局部改写方向。
- 必要时给出 1-2 句示例改写。

不允许：

- 直接输出整篇改写稿。
- 为了追求传播性改变文章核心结论。
- 把文章改成教程、面试题库或学习路线。
- 用夸张标题、焦虑表达或营销式话术提高吸引力。
