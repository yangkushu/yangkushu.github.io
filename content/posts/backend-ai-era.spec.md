完整版发布到 GitHub 个人站、知乎和公众号。

小红书版本等完整版定稿后，再拆分为多篇笔记；当前阶段不提前规划具体标题和提纲，避免影响完整版结构。

## 读者定位

核心读者是正在求职或准备跳槽的后端开发，尤其是有一定经验、以 Java / Go 等技术栈为主、没有明显大厂或算法背景的人。

外延读者是所有关心 AI 时代后端定位变化、想提前补齐大模型应用能力的后端开发。

## 文章定位

主线是一次后端求职复盘。

副线是从这次经历中提炼出普通后端在 AI 时代的转向建议。

文章不是宏观就业市场分析，也不是大模型应用后端学习路线、RAG / Agent 教程或面试题库。技术内容只服务于求职复盘，写到能解释岗位匹配和准备方向即可。

## 核心结论

这篇文章的底层判断是：AI 时代后端岗位的变化，不只是岗位数量变化，也不是传统后端内容消失，而是后端基本盘仍然存在，但 AI 相关能力在面试和工作方式里的比重正在增加。

传统后端能力的考察没有消失，也不应写成被 AI 取代。数据库、并发、系统设计、稳定性、线上问题排查等传统后端内容仍然是基础。

在传统后端能力之上，AI Coding 正在成为新的基础能力。很多后端岗位已经默认候选人会使用 AI Coding，并会关注候选人如何用 AI 提效、约束生成质量、审查代码和验收结果。

大模型应用能力是另一个层级：懂 RAG / Agent / 工具调用，能把大模型能力接入业务系统，对于普通后端是明显加分项；进一步发展，则可以对应到“大模型应用后端 / 大模型应用全栈”这类岗位。

从个人求职经历看，普通后端想靠近 AI，不一定要硬转算法或纯 AI 开发。更现实的路径，是把已有后端工程能力迁移到大模型应用落地中，优先关注“大模型应用后端 / 大模型应用全栈”这类岗位。

这不是说所有后端都必须转向大模型应用后端，也不是说传统后端没有价值。只是从这次求职经历看，如果一个普通后端想靠近 AI，又不具备算法和模型背景，那么大模型应用后端是一条更现实的路径。

## 表达边界

文章基于个人经历和阶段性判断，不写成确定性的行业结论。

不要渲染焦虑：

- 不使用“后端已死”“不转 AI 就淘汰”这类表达。
- 不把 AI 描述成单向取代程序员，而是描述成改变岗位要求和开发方式。
- 不贬低传统后端经验，而是强调后端经验在 AI 应用落地中的迁移价值。
- 结尾落到可操作建议和重新定位，而不是焦虑情绪。

关于岗位变化，重点不是写普通后端岗位减少，也不是写 AI 简单取代后端，而是写传统后端内容仍然是基本盘，AI 相关能力在面试和工作方式里的比重正在增加：传统后端能力仍然要考察；AI Coding 逐渐成为基础能力；懂大模型应用和业务落地能力会成为重要加分项。公司对“纯业务开发 / 纯编码执行”的需求更谨慎，对候选人的综合能力要求更高。

关于 AI 和后端职责，不要写成“AI 做不了系统设计 / 技术选型 / 问题排查”。更稳妥的表达是：AI 已经能参与编码、方案讨论、技术选型、问题排查等工作，但在真实项目里，后端仍然要对系统结果负责。比如需求边界是否清楚、方案是否适合当前业务、并发和稳定性风险有没有覆盖、线上问题如何闭环、AI 生成的代码能不能上线，这些判断和责任目前仍然离不开开发者。

## 开头要求

开头不要以“工作不好找”“普通后端岗位变少”这类大环境共识作为第一句，也不建议在开头主动强调岗位减少。这类表达在当前环境下信息增量不够，容易显得像废话。

第一句应从“求职复盘后的判断”切入，而不是从具体时间点切入。避免使用“年初找工作时”作为开头，因为这会让读者感觉事情已经过去一段时间，削弱文章的当下价值。

第一句要表达的核心不是“后端内容变了”，也不是泛泛地说“考察范围变宽了”，而是：后端面试并没有少考后端本身，但 AI Coding 和大模型应用相关问题正在更频繁地进入面试评价。

第一句可以采用类似结构：复盘这次求职，我最明确的感受是：后端面试并没有少考后端本身，但 AI Coding 和大模型应用相关问题出现得更频繁了。

开头应尽快引出核心问题：AI 时代后端岗位的默认能力正在扩展，AI Coding 正在成为基础能力，懂大模型应用则是重要加分项，并可能进一步发展为“大模型应用后端”方向；后端需要重新理解自己的定位。

开头控制篇幅，避免反复铺陈求职困难。个人背景、求职体感和核心结论要尽快进入主线。

## 文风要求

文章定位为技术经验复盘，语气要克制、清晰、具体。

可以保留个人经历和第一人称，但避免过度口语化、情绪化和泛泛表达。少用“没以前好找了”“压哨”“硬冲”“往 AI 靠”等偏口语词，除非确实能增强表达。

优先使用具体观察、面试问题、岗位要求变化和技术落地点来支撑观点。

## 岗位分类口径

正文保持两类讨论：

- AI 开发：更偏算法、模型、NLP、微调、训练、评估等能力。
- 大模型应用后端：以业务系统和工程落地为主，叠加 Agent、RAG、工具调用、AI Coding 等能力。

现实 JD 里的“大模型应用开发 / AI 应用开发 / 大模型全栈”等名称边界很模糊，可能同时包含应用开发、后端、前端和部分算法能力。文章重点不是做岗位分类，而是说明传统后端更容易迁移到哪一类工作内容。

## 关键观点

### 后端岗位要求变化

这次找工作最明显的体感是，后端岗位的基本盘没有变，但对 AI 相关能力考察的比重正在增加。传统后端内容没有减少，仍然会考数据库、并发、系统设计、稳定性、线上问题排查等能力。在此基础上，很多岗位会增加 AI Coding 相关考察，关注候选人如何用 AI 提效、约束生成质量、审查代码和验收结果。

大模型应用能力需要和 AI Coding 区分开：AI Coding 更像普通后端也需要补齐的新基础能力；RAG / Agent / 工具调用 / 大模型接入业务流程，则更像加分项和进一步的岗位分支。

### 传统后端经验仍然有迁移价值

后端不是从零开始转 AI。大模型应用最后仍然要落到业务系统里：权限、数据、流程、接口、任务调度、监控、部署、稳定性、成本控制，这些都离不开后端经验。

所谓转向大模型应用后端，本质是把原来的工程经验迁移到新的业务形态里。

### AI 开发不适合普通后端短期硬冲

可以讲自己也尝试过更偏 AI 开发的岗位，但要控制篇幅，并保持提醒语气。

主要原因是学历门槛、项目经验、知识背景不匹配。很多岗位会看算法、模型、NLP、微调、训练、评估等背景，这和传统后端工程经验不是同一条积累路径。对普通后端来说，短期硬冲这类岗位，投入产出比可能并不理想。

这段不要写成失败叙事，也不要写成对岗位的批判，而是提醒读者不要只看岗位名，要看 JD 和自身匹配度。

### 大模型应用后端首先仍然是后端岗位

大模型应用后端不是只会调 API，也不是只会写 Prompt，而是在后端系统中接入和落地 AI 能力。

这类岗位仍然考察后端基本功，不因为岗位带 AI 就降低要求。AI 相关能力是在后端能力之上的增量，而不是对后端能力的替代。

### 能力建议要简练

能力部分只写到“求职复盘能支撑的程度”，不展开成系统学习路线。

重点保留三组：

- 后端基本功不能丢：面试仍然会考后端，不因为岗位带 AI 就降低要求。
- RAG / Agent 要有项目感：不仅知道概念，还要能讲流程、选型、问题处理。
- AI Coding 要会落地：能说明自己怎么用 AI 提效、怎么约束质量、怎么验收结果。

### 补齐项目经验

这部分有必要，但只写一小段。

如果没有真实公司项目，至少要准备一个小而完整的 RAG / Agent 项目，用来证明自己理解完整链路。重点不是项目多大，而是能讲清楚场景、流程、选型、问题和评估。

不展开实现细节。

### 面试问题只做经历佐证

面试问题不要整理成题库，也不要归类成系统面试指南。

保留“我被问过这些问题”的口吻，用少量真实问题说明：这类岗位更看重项目经验、场景判断和排查思路，而不是背概念。

可保留的问题方向：

- Agent 怎么避免死循环？
- 工具调用失败怎么处理？
- RAG 效果不好时从哪里排查？
- AI Coding 用过哪些 Skills，分别在什么场景解决了什么问题？

### AI Coding 是独立但次一级的必备能力

AI Coding 不是文章主线，但要作为一个明显的小节。

需要表达几个点：

- 面试过程中逐渐发现，很多公司已经默认候选人会使用 AI Coding。
- 被问的重点不是“你用不用工具”，而是“你怎么用、在哪些场景用、解决过什么问题”。
- 入职后进一步确认，AI Coding 已经成为实际开发中的主要工作方式之一。
- AI Coding 不等于让 AI 写代码，而是人负责拆任务、定边界、做架构判断、审查代码和验收结果。
- AI 仍然会遗漏工程问题，例如集群部署并发、资源争抢、边界条件、质量控制等。

## 求职细节边界

写体感和阶段性结论，不写具体数字。

可以写：

- 年初找工作。
- 后端岗位的基本盘没有变，传统后端内容仍然会考，对 AI Coding 相关内容考察的比重正在增加。
- 大模型应用能力对于普通后端是加分项，也可以进一步对应到大模型应用后端岗位。
- 尝试过 AI 开发方向但匹配度不高。
- 后来把方向收敛到大模型应用后端。
- 面试中发现后端基本功、RAG / Agent 项目经验、AI Coding 都会被关注。
- 五一前入职。

不建议写：

- 投了多少份简历。
- 收到多少面试。
- 哪些公司。
- 薪资变化。
- 具体 offer 数量。
- 过于细的失败经历。

## 建议结构

1. 开头从年初找工作切入，说明后端岗位的基本盘没有变，传统后端内容仍然会考，但对 AI Coding 等 AI 相关能力考察的比重正在增加。
2. 引出自己的问题：后端在 AI 时代该怎么重新定位。
3. 说明自己曾经观察或尝试过 AI 开发方向，但学历门槛、项目经验和知识背景匹配度不高。
4. 转到大模型应用后端：它更能复用后端经验，也能进入 AI 应用落地场景。
5. 解释大模型应用后端不是只调 API，而是在后端系统里落地 AI 能力。
6. 简要说明需要补齐的能力：后端基本功、RAG / Agent 项目感、AI Coding。
7. 用少量真实面试问题作为佐证，不写成题库。
8. 在 AI Coding 小节加入入职后的验证：它已经成为实际开发中的主要方式之一。
9. 结尾回扣“重新定位”：不是从后端跳到算法，也不是只学几个框架，而是把已有工程经验迁移到 AI 应用落地中，同时补齐新的应用能力。

## 结尾方向

结尾要接应标题里的“重新定位”。

推荐落点：对普通后端来说，AI 时代不是简单地从后端跳到算法，也不是只靠多学几个框架就能解决问题。更重要的是重新理解自己的位置：把已有的工程经验迁移到 AI 应用落地里，同时补齐 RAG、Agent、AI Coding 这些新能力。

最后保留个人边界：这只是这次求职后的阶段性判断，但至少对自己来说，它比硬转纯 AI 开发更现实。

## 读者评审流程

完整版初稿完成后，可以让 AI 以读者视角进行评审。

读者评审采用 2+1 设计：两个必选目标读者，一个终稿前专业审稿人。

- 初稿和大改后：使用读者 A、读者 B。
- 准备发布前：使用读者 A、读者 B、读者 C。
- 读者 C 只做专业性兜底，不决定文章主线。

### SubAgent 评审方式

读者评审由主 agent 启动 subAgent 执行。

主 agent 负责从本文 spec 中裁剪当前读者需要的评审规则，并将裁剪后的规则发送给 subAgent。裁剪内容包括：当前评审阶段、共用检查项、AI 味检查标准、当前读者角色定位、当前读者主要判断、当前读者重点打分、当前读者不应该要求、反馈格式、当前读者相关通过标准、评审边界和唯一文章路径。

subAgent 不自行读取 spec，避免受到其他读者定义、旧流程或无关规则影响。主 agent 不向 subAgent 提供其他读者的完整定义、其他读者的评审结果，或主 agent 对文章质量的预判。

文章正文 `content/posts/backend-ai-era.md` 是唯一评审对象，允许 subAgent 自行读取。subAgent 只允许读取该文件；不读取 spec、README、git 历史、其他文章或项目文档；不修改文件，不运行构建，不参考其他读者反馈。

subAgent 只输出评审结果，不做采纳决策；是否修改正文由主 agent 汇总后再由作者确认。

### 共用检查项

所有读者都需要关注：

- 文章是否制造“后端不转 AI 就会被淘汰”的焦虑。
- 文章是否围绕求职复盘和重新定位展开，而不是变成教程、题库或学习路线。
- 文章是否存在明显 AI 味。

AI 味重点检查以下信号：

- 模板化结构明显，段落像通用文章生成模板。
- 泛化判断过多，缺少个人经历、面试问题或真实场景支撑。
- 求职复盘的现场感不足，看不到作者判断变化的过程。
- 表达过度平滑，只有正确结论，缺少具体取舍。
- 热词堆叠，RAG / Agent / AI Coding 等概念没有落到场景。
- 缺少作者自己的犹豫、试错和修正过程。

如果 `AI 味控制` 低于 4/5，读者必须指出最像 AI 生成内容的 1-3 个具体位置，并说明原因。

- 工程真实感：是否符合真实项目落地经验。
- 概念克制：是否避免概念堆砌和过度展开。
- AI 味控制。

不应该要求：

- 不要求文章面向专家读者重写。
- 不要求文章补充大量技术细节。
- 不要求把文章改成教程、技术方案或白皮书。

### 反馈格式

每个读者使用统一反馈结构：

```md
## 结论

是否建议继续修改：是 / 否

## 打分

- 维度一：x/5
- 维度二：x/5
- 维度三：x/5
- AI 味控制：x/5

## 主要问题

1. ...
2. ...
3. ...

## 最有效的部分

...

## 最需要修改的部分

...

## 具体建议

1. ...
2. ...
3. ...
```

### 通过标准

- 两个必选读者的 `AI 味控制 < 4/5` 时，必须修改。
- 读者 A 的 `焦虑控制 < 4/5` 时，必须修改。
- 读者 B 的 `处境贴合 < 4/5` 或 `经验迁移 < 4/5` 时，必须修改。
- 终稿前读者 C 的 `技术准确性 < 4/5` 或 `岗位边界 < 4/5` 时，必须修改。
- 单项 3/5 不一定全盘否定，但需要指出具体问题位置和原因。

### 评审边界

读者评审角色只负责反馈问题，不直接重写全文。

允许：

- 指出具体问题位置。
- 说明为什么影响目标读者、技术可信度或阅读体验。
- 提供局部改写方向。
- 必要时给出 1-2 句示例改写。

不允许：

- 直接输出整篇改写稿。
- 为了追求传播性改变文章核心结论。
- 把文章改成教程、面试题库或学习路线。
- 用夸张标题、焦虑表达或营销式话术提高吸引力。
