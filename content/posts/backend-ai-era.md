这次找工作让我体会到，对于普通后端来说，AI Coding 和大模型应用能力，确实会影响面试机会。

这里的“机会”，不是说传统后端能力不重要了。后端基本功（八股）仍然是考察重点。

变化在于，很多岗位还会继续看你有没有 AI Coding 的实际使用经验。大模型应用相关岗位还会看你是否做过 RAG / Agent 项目，能不能讲清楚一个大模型应用从场景、流程、选型到问题处理和评估的过程。

这篇文章想回答的问题更具体：一个没有算法和模型背景的普通后端，在 AI 时代应该怎么重新定位。

这次求职里，我后来把 AI 相关要求分成了两层看。一层是 AI Coding，它更像后端也要补齐的日常开发能力；另一层是具体岗位方向，主要是偏模型、算法或复杂应用的 AI 开发，以及偏工程接入的大模型应用后端。

我自己一开始也尝试过 AI 开发方向，后来逐渐把重点放到大模型应用后端，并在五一前入职了相关岗位。下面主要想讲清楚为什么我认为这个方向对普通后端更合适。


### AI 开发
这里说的 AI 开发，包括一些“大模型开发”岗位。它们有的更偏大模型基础、NLP、微调、训练、评估，有的虽然也做 RAG / Agent，但更关注模型效果、评估和调优。

这也是我找工作时困惑比较久的一点：大模型开发和大模型应用后端看起来很像，都会出现 RAG、Agent、大模型应用这些词，但实际要求并不一样。前者通常更看重 AI 相关背景、项目深度和技术判断；后者首先还是后端岗位，更看重你能不能把 RAG / Agent 做成后端系统的一部分，和原有的数据、权限、监控等体系配合起来。

我自己尝试下来，偏 AI 开发的岗位有效面试很少。最直接的判断信号来自岗位描述：要求里不只是会接入模型 API，而是会写到大模型基础、NLP、微调、训练、评估等经验。还有一些岗位标题写着“大模型开发”，但实际内容更接近复杂的大模型应用系统，面试关注点也会落到 RAG / Agent 的项目深度、效果评估和调优经验上。

这些要求和传统后端工程经验不是同一套知识体系。主要问题不是这个方向不好，而是学历门槛、项目经验和知识背景都不太匹配。

所以对普通后端来说，短期内只靠补几个框架、背一些概念去投这类 AI 开发岗位，回报未必明显。现在很多大模型相关岗位的标题并不统一，判断岗位是否适合自己，不能只看岗位名，还是要看岗位描述里的实际工作内容。

### 大模型应用后端
我后来把重点转移到了大模型应用后端。这里的“大模型应用后端”，不是只会调用大模型 API 或者写 Prompt，而是把 RAG、Agent 等能力接入后端系统，用来解决具体业务问题。

一个大模型应用通常仍然要面对权限控制、数据流转、任务调度等问题。模型能力只是系统的一部分，后端仍然要负责把它接入原有业务系统，并保证问题能被观察、能被排查。

这类岗位首先仍然是后端岗位，对后端的考察和传统后端岗位没有本质区别。Java / Go 语言和框架、数据库、中间件、系统设计等内容仍然是重点。

对普通后端来说，如果想面试这类岗位，至少要准备一个能讲清楚的 RAG / Agent 项目。即使不是公司里的正式项目，也最好能围绕一个真实问题做完整：数据从哪里来，模型能力放在哪里，异常和效果问题怎么处理。

这里真正能迁移的，是后端长期处理系统问题的经验。比如 Agent 的多轮调用，如果只把它理解成“让模型自己规划任务”，面试里很容易讲浅。它进入后端系统之后，就会变成任务状态怎么保存、工具调用失败后怎么处理、重复执行会不会造成脏数据这类问题。

RAG 也是类似的。把文档向量化再检索，只是最容易演示出来的部分。真正被追问时，问题往往会回到权限过滤、检索耗时、结果不准、Bad Case 怎么收集这些工程细节上。

所以面试里聊 RAG / Agent 时，我发现对方通常不会只停留在概念层面。这类面试通常会先问项目的业务架构，然后围绕项目经验、场景判断和排查思路展开。比如我被问过：

- Agent 如何避免死循环？
- 工具调用失败怎么处理？
- RAG 效果不好时从哪里排查？

这些问题没有标准答案。它们真正区分的是，你是在讲概念，还是在讲一个实际会运行、会出错、需要排查的系统。

其中 “Agent 如何避免死循环” 这个问题很典型。它表面上是在问 Agent，实际追问的是你有没有把模型调用当成一个会失败、会重复、会失控的系统组件来处理。只回答“限制最大轮数”是不够的，因为最大轮数只能终止一次调用，不能解释任务为什么卡住、卡住后状态怎么恢复、失败时如何降级，以及日志里要留下哪些信息方便排查。

### AI Coding：面试里更关心你怎么用
这次面试过的岗位中，AI Coding 出现的频率也很高。很多公司关心的已经不是“你会不会用 AI 写代码”，而是“你怎么用、在哪些场景用、解决过什么问题”。

入职后我对这一点的感受更明显。AI Coding 不是单独拿一个工具生成代码，而是会进入任务拆分、方案讨论、编码、代码审查和测试验收这些环节。它能把很多局部编码工作做得很快，但需求边界、系统风险和上线判断仍然要由开发者负责。

举个实际遇到的例子：让 AI 写一个处理数据的简单定时任务，它可以很快完成编码工作。但如果没有明确提醒，它可能不会主动考虑服务集群部署下的并发问题，也可能漏掉同一条数据被多个任务同时处理时的资源争抢问题。

这些问题，并不是换一个更好的模型就一定能完全避免。我现在更习惯先把任务边界、并发约束、异常处理和验收方式写清楚，再让 AI 参与实现。生成结果出来后，重点看它有没有漏掉边界条件、有没有破坏项目原有规范，必要时再补测试或手工验证。

面试里聊到 AI Coding 工具时，我也发现只说“用过 Cursor、Claude Code”意义不大。对方更关心的是我怎么管理规则、上下文、自定义指令和项目规范，在哪些场景把它们放进开发流程，以及最后如何审查和验收结果。

### 重新定位
现在再看这次求职，我会更关注岗位描述里的实际系统要求：AI 能力接在哪个业务流程里，数据和权限怎么处理，效果不好时谁来排查，系统上线后谁对稳定性负责。

这是我目前对“重新定位”的理解。对普通后端来说，现阶段不一定要硬转模型训练、微调或算法岗位；更现实的方向，是把原来的后端工程经验放到大模型应用里，同时补上 RAG、Agent 和 AI Coding 这些新的应用能力。这个判断以后可能还会调整，但至少这次求职下来，它比只看岗位名去追“大模型开发”更清楚。
