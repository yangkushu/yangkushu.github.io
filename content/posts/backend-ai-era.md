这次找工作让我感觉，对于普通后端来说，大模型应用，确实是一个非常好的发展方向。
在当前大环境下，这也是不多的还有很多新岗位的方向了。

这里不是说传统后端能力不重要了。后端基本功（八股）仍然是考察重点。
除了后端，相关岗位还会考察大模型相关的 RAG / Agent 经验，能不能讲清楚大模型应用从场景、流程、选型到问题处理和评估的完整过程。

这篇文章我想更具体的谈谈我的看法：没有算法和模型背景的普通后端，在 AI 时代应该怎么重新定位。

我把这次求职里接触到的大模型相关岗位先粗略分成两类：“大模型开发”、“大模型应用后端”。
- “大模型开发”更偏模型、算法，或者更复杂的大模型应用开发；
- “大模型应用后端”则是在后端系统里使用 RAG、Agent 等能力。

最后再聊聊现在热门的面试话题：AI Coding。


### 大模型开发
这里说的“大模型开发”，包括很多类似的岗位，现在称呼也比较乱，比如“大模型开发”，“AI 开发”等等。但是可以看到他们的岗位描述中一般都包含大模型基础、NLP、微调、训练、评估。有的虽然也做 RAG / Agent，但更偏复杂的大模型应用开发，比如围绕模型能力做应用设计、效果评估和调优。

我自己一开始也主要往“大模型开发”方向投简历，但是“大模型开发”的岗位虽然很多，但是面试很少。
主要问题不是这个方向不好，而是学历门槛较高，需要的项目经验和知识背景也不太匹配。很多岗位会看大模型基础、NLP、微调、训练、评估等经验，这些和传统后端工程经验不是同一套知识体系。

所以对普通后端来说，短期内只靠补几个框架、背一些八股文去投这类“大模型开发”岗位，投入产出比可能并不理想。这类岗位可能在学历比较好的情况下，大量投入精力进行准备，才有比较好的面试机会。

所以我后来逐渐把重点放到“大模型应用后端”。
### 大模型应用后端

“大模型应用后端”这个岗位名字和刚才说的“大模型开发”很像，岗位描述上都会出现 RAG、Agent、大模型这些关键词。但实际上有如下不同：
- 一般不会考察大模型基础、微调、训练等知识。主要会考察 RAG / Agent 的知识和实际使用经验。
- 除了考察后端的内容外，更看重你能不能把大模型应用做成完整后端系统的一部分。
所以我认为这个方向对普通后端更合适。

不要小看好像只有几个关键字的区别，找工作时面对这些刚接触的岗位，这之间的区别也让我困惑了很久。

我后来把投简历的重点转移到了“大模型应用后端”，面试就明显多了很多，offer 也有了。

下面来具体讲讲我对这个岗位的理解，以及下面试下来的经验。
需要再次强调的是，这里的“大模型应用后端”，不是只会调用大模型 API 或者写 Prompt，而是把 RAG、Agent 等能力接入后端系统，用来解决具体业务问题。
一个大模型应用通常仍然要面对权限控制、数据流转、任务调度等问题。模型能力只是系统的一部分，后端仍然要负责把它接入原有业务系统，并保证系统的可观测性和可用性。
这类岗位对后端的考察和之前相同。Java / Go 语言和框架、数据库、中间件、系统设计等内容仍然是重点。

如果想面试这类岗位，必须要准备一个 RAG / Agent 项目。我认为可以准备一个以 LangGraph 框架实现，包含 RAG、Multi-Agent 的一个大而全的项目。
- RAG 部分需要掌握文档处理、向量化、检索、Rerank、结果生成、评估这一套完整流程，并能说明常见问题的处理、整体效果的优化。
- Agent 部分需要能说明项目里的 Multi-Agent 的调度、工具调用、状态流转、失败处理、循环控制等流程。
  
面试里通常会先问项目的业务架构，然后围绕关键技术点、常见错误处理和输出效果优化等具体问题展开提问。比如下面几个问题我被问过好几次：
- Agent 如何避免死循环？
- 工具调用失败如何处理
- 如何提升 RAG 的效果？

这些问题没有标准答案，关键是要能结合具体项目和场景回答。

### AI Coding 正在成为必备能力
这次面试过的岗位中，AI Coding 被问到的次数越来越多，后来已经成为重点之一。现在面试中，公司关心的已经不是“你会不会用 AI 写代码”，而是“你怎么用 AI 写好代码”。

AI Coding 目前已经成为实际开发中的主要工作方式。目前已经从 Vibe Coding 进化到 Harness Engine。它是把 AI Coding 纳入开发流程：做架构判断、人制定边界、审查代码和验收结果。

AI 可以参与计划、编码、方案讨论和问题排查，但真实项目最后仍然要有人对系统结果负责。需求边界是否清楚，方案是否适合当前业务，并发和稳定性风险有没有覆盖，AI 生成的代码能不能上线，这些判断不能完全交给工具。

目前，AI 编程还无法达到“全自动”的水平。AI 多多少少还是会有一些不太靠谱的地方，你甚至不知道它什么时候就在你的代码里留了个“坑”。

举个实际遇到的例子：让 AI 写了一个处理数据的定时任务，它很快的完成编码工作。但它没有考虑服务集群部署下的并发问题，也漏掉了同一条数据被多个任务同时处理时的资源争抢问题。
这些问题，并不是换一个更好的模型就一定能完全避免。更重要的是开发者自己知道哪些地方容易出问题，并且能通过开发规范、代码审查和测试验收等手段控制风险。

AI Coding 相关面试的考察重点也是在“实操”上，包括 Harness Engine 相关概念如何落地，你的项目中完整的 AI Coding 开发流程是什么样的，如何防止代码质量滑坡等。

我在面试中还被问到了好几次的问题：“你用过哪些 Skills？”
在回答这个问题时，不能只说“我用了哪些 Skills”，而要结合实际场景回答：我在哪些场景用过哪些 Skills，解决了什么问题，最后如何验收结果。

### 重新定位
所以，我选择的后端在 AI 时代的新的方向，是把已有的工程和行业经验用到大模型应用开发里，同时补齐 RAG、Agent、AI Coding 这些新能力，同时还保留传统的后端工程能力。

以上这只是我这次求职后的阶段性判断，但至少对很多普通后端来说，“大模型后端应用”这个岗位比转“大模型开发”更现实，也更能沿用原来积累下来的后端经验。
