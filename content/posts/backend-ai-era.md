这次找工作让我体会到，对于普通后端来说，AI Coding 和大模型应用能力，确实能明显带来更多面试机会。

这里的“机会”，不是说传统后端能力不重要了。后端基本功（八股）仍然是考察重点。

变化在于，很多岗位还会继续看你有没有 AI Coding 的实际使用经验，是否做过 RAG / Agent 相关项目，能不能讲清楚大模型应用从场景、流程、选型到问题处理和评估的完整过程。

这篇文章想回答的问题更具体：一个没有算法和模型背景的普通后端，在 AI 时代应该怎么重新定位。

我把这次求职里接触到的 AI 相关能力先粗略分成三类：AI Coding、AI 开发、大模型应用后端。AI Coding 更像后端也需要补齐的新能力；AI 开发更偏模型、算法，或者更复杂的大模型应用开发；大模型应用后端则是在后端系统里使用 RAG、Agent 等能力。

我自己一开始也尝试过 AI 开发方向，后来逐渐把重点放到大模型应用后端，并在五一前入职了类似岗位。下面主要想讲清楚为什么我认为这个方向对普通后端更合适。


### AI 开发
这里说的 AI 开发，包括一些“大模型开发”岗位。它们有的更偏大模型基础、NLP、微调、训练、评估，有的虽然也做 RAG / Agent，但更偏复杂的大模型应用开发，比如围绕模型能力做应用设计、效果评估和调优。

这也是我找工作时困惑比较久的一点：大模型开发和大模型应用后端看起来很像，都会出现 RAG、Agent、大模型应用这些词，但实际要求并不一样。前者通常更看重 AI 相关背景、项目深度、相关项目经验和技术判断；后者首先还是后端岗位，更看重你能不能把 RAG / Agent 做成完整后端系统的一部分，和原有的数据、权限、监控等体系配合起来。

我自己尝试下来，偏 AI 开发的岗位有效面试很少。主要问题不是这个方向不好，而是学历门槛较高，需要的项目经验和知识背景也不太匹配。很多岗位会看大模型基础、NLP、微调、训练、评估等经验，这些和传统后端工程经验不是同一套知识体系。

所以对普通后端来说，短期内只靠补几个框架、背一些概念去投这类 AI 开发岗位，投入产出比可能并不理想。现在很多大模型相关岗位的标题并不统一，判断岗位是否适合自己，还是要看岗位描述里的实际工作内容。

### 大模型应用后端
我后来把重点转移到了大模型应用后端。这里的“大模型应用后端”，不是只会调用大模型 API 或者写 Prompt，而是把 RAG、Agent 等能力接入后端系统，用来解决具体业务问题。

一个大模型应用通常仍然要面对权限控制、数据流转、任务调度等问题。模型能力只是系统的一部分，后端仍然要负责把它接入原有业务系统，并保证系统的可观测性和可用性。

这类岗位首先仍然是后端岗位，对后端的考察和之前相同。Java / Go 语言和框架、数据库、中间件、系统设计等内容仍然是重点。

对普通后端来说，如果想面试这类岗位，至少要准备一个能讲清楚的 RAG / Agent 项目。

RAG 部分不能只停留在“把文档向量化再检索”。需要掌握文档处理、向量化、检索、Rerank、结果生成、评估这一完整流程，并能说明常见问题的处理、整体效果的优化，以及相关技术方案的取舍。

Agent 部分需要能说明工具调用、状态流转、失败处理、循环控制等问题在项目里如何设计和处理。

这类面试通常会先问项目的业务架构，然后围绕项目经验、场景判断和排查思路等具体问题展开提问。比如我被问过：

- Agent 如何避免死循环？
- 工具调用失败怎么处理？
- RAG 效果不好时从哪里排查？

这些问题没有标准答案，关键是能结合具体项目和场景回答，说明自己真的做过完整过程，而不是只看过概念。

### 一项必备能力 - AI Coding
本次我面试过的岗位中，AI Coding 出现的频率也很高。很多公司关心的已经不是“你会不会用 AI 写代码”，而是“你怎么用、在哪些场景用、解决过什么问题”。

入职后我也进一步确认，AI Coding 已经成为实际开发中的主要工作方式之一。它不是让 AI 代替程序员，而是把 AI 纳入开发流程：人负责拆任务、定边界、做架构判断、审查代码和验收结果。

AI 可以参与编码、方案讨论和问题排查，但真实项目最后仍然要有人对系统结果负责。需求边界是否清楚，方案是否适合当前业务，并发和稳定性风险有没有覆盖，AI 生成的代码能不能上线，这些判断不能完全交给工具。

举个实际遇到的例子：让 AI 写一个简单处理数据的定时任务，它可以很快完成编码工作。但如果没有明确提醒，它可能不会主动考虑服务集群部署下的并发问题，也可能漏掉同一条数据被多个任务同时处理时的资源争抢问题。

这些问题，并不是换一个更好的模型就一定能完全避免。更重要的是开发者自己知道哪些地方容易出问题，并且能通过任务拆分、上下文约束、代码审查和测试验收控制风险。

面试里还有一个很容易被问到的问题：“你用过哪些 Skills？”这个问题不能只回答“我用了哪些 Skills”，而要结合实际场景回答：我在哪些场景用过哪些 Skills，解决了什么问题，最后怎么验收结果。

所以，我现在对后端在 AI 时代的重新定位，大概是这样的：不是从后端跳到算法，也不是只学几个大模型框架，而是把已有的工程经验用到 AI 应用开发里，同时补齐 RAG、Agent、AI Coding 这些新能力。

这只是我这次求职后的阶段性判断，但至少对普通后端来说，它比短期直接转纯 AI 开发更现实，也更能接住原来积累下来的后端经验。
